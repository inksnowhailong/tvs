## 架构规范

在生成代码时，必须严格遵守以下架构规范。这些规范确保代码结构清晰、可维护、可扩展。生成前，先分析需求，确保设计符合规范；若冲突，调整设计并解释。

### 单一职责原则 (SRP)
每个模块或组件只负责一类特定的事务，且所有相关逻辑必须集中在此模块中。避免让模块处理不属于其职责范围的任务。
- 示例：
  - 全局共享状态（如token验证、环境标识、国际化语言）应放置在核心层（core）的专用状态管理模块中（例如，使用Pinia的store）。UI视图层（views）中的组件应引用核心层的状态，并转换为其内部状态，而非直接在views中重新实现。
  - 登录功能应在核心层的专用登录模块中实现。不同UI组件（如views中的登录页面）仅调用此模块的能力，而非在UI层重复编写登录逻辑。

### 开闭原则 (OCP)
优先通过扩展现有实现来满足新需求，而非修改原有代码。设计时，确保逻辑易于扩展；如果必须修改，限制影响范围。通过层级划分，使高层组件不受低层修改影响。
- 示例：
  - 添加新功能时，创建子类或新模块扩展现有接口，而不是直接编辑核心实现代码。
  - 如果修改低层模块（如HTTP请求适配器），确保高层（如UI用例接口）不受影响，通过抽象接口隔离变化。

### 依赖反转原则 (DIP)
这是系统中最重要的原则！模块间依赖应指向抽象接口而非具体实现。首先设计高层抽象类型来定义依赖关系，然后在低层实现具体类型。修改抽象接口时，应明确影响的具体实现，但修改实现不应反过来影响高层依赖。
- 示例：
  - 高层模块（如业务用例）依赖抽象接口（如`ILoginService`），低层模块（如具体登录实现）实现此接口。
  - 业务领域按领域划分模块（如用户模块、订单模块），并明确高层（抽象）和低层（实现）划分。源代码依赖始终指向抽象。

### 层级划分
**横向**：最高层为全抽象层（实体接口、用例接口、业务服务模块接口、全局功能模块接口等）；中间为实现层（不同业务领域模块，包含各自业务逻辑、UI响应式状态信息、UI交互专用函数等）；最低为UI层（依赖抽象层调用，通过依赖注入引入实现层具体对象）。
**纵向**：按业务领域划分模块（如用户领域），包括纯功能模块（如登录、HTTP）。
编程细分：
- 抽象层：实体（如IUserEntity）、用例（如ILoginUseCase）、服务（如ILoginService）。
- 实现层：业务逻辑实现、状态管理（如Pinia store for UI reactivity）、交互函数。
- UI层：视图组件，注入实现（如通过DI容器），调用抽象接口。


在生成代码时：
- 如果生成完整系统，确保core层包含共享逻辑，views层仅处理UI交互。
- 始终验证生成的代码是否遵守所有原则；如果违反，重新设计并说明优化。
